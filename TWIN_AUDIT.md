# TWIN — Full App Audit (2026-07-07)
_Multi-agent research pass: 8 specialist agents read the real codebase + 1 synthesis. 58 findings._

## Verdict
TWIN is a stable, well-built shell that repeatedly computes real intelligence and then throws it on the floor: twin_score() returns a score, color, ring offset, named verdict, and a next-action, and home.html even ships the CSS for a full hero card — but the content block renders none of it, showing only a 17px number. The single biggest gap is not missing features; it's that the app already does the 'aha' work server-side and never puts it on screen. Compounding that, the only return mechanic (bell/pop) is driven by social counts that will be near-zero in a friends-and-family beta, so nothing pulls a lonely tester back. Fix the wiring (render the score/streak/memory the app already has) and add one unprompted return channel (Resend evening email) before touching trackers — the vessel is empty mostly because its own outputs are disconnected, not because the outputs don't exist.

## Top 3 highest-leverage moves
1. Render the already-computed Twin Score hero on home.html — the ring (twin.offset/color), twin.verdict headline, and twin.action block all have dead CSS at home.html:156-167 and JS at line 392 waiting for markup. Zero new data; this alone converts Home from empty vessel to daily payoff screen.
2. Build the only pre-push return loop: a streak-at-risk hero on Home using the already-computed-but-unreferenced `shielded` var + u.streak ('Your 12-day streak ends at midnight'), plus a once-daily Resend evening email (send_email is already wired for signup codes) to users who haven't checked in. Social-count notifications are dead in a F&F beta.
3. Close the memory loop cheaply: pass the morning CheckIn.reflection into checkout ('You said today's #1 was X — done?'), pass last night's 'Tomorrow:' note into the morning hint, and surface one server-side 'AXON noticed' insight (e.g. mood on <6h vs 7h+ sleep) from existing check-in history. This is the strongest 'it knows me' proof achievable with no trackers.

## Quick wins (buildable now, high impact)
### Render the Twin Score hero on Home
- **Why:** twin_score(cu) is passed to home.html but only a tiny greet-bar chip uses it; the .tscore ring + .tscore-verdict + .tscore-action CSS (home.html:156-167) and the .tscore-fg JS (line 392) are dead. The app's single most intelligent output is invisible.
- **How:** Add the hero markup under the greeting consuming the existing `twin` object (score, color, offset, verdict, action). Add a greyed '—' ring empty-state when twin.has is False with a gold 'Check in to unlock your first Twin Score' CTA to /checkin. Pure wiring, S effort.
### Render a 7-dot 'this week' streak row on Home
- **Why:** stats_ctx already produces s['checked'] (Sun→Sat booleans) and home.html already ships .week-bars/.wb CSS, but nothing renders it. The only streak surface is a bare flame number with no chain to protect.
- **How:** Loop s['checked'] into 7 dots in the home content block, today's slot empty and pulsing until checked in. Zero new data; reuses existing CSS.
### Kill the fake trend numbers (biggest 'demo' tell)
- **Why:** analytics.html:68-92 prints hardcoded literals ('+0.4 from last week' etc.) and domain trends are hardcoded '+0' at app.py:1196-1201. A first-time friend sees fabricated week-over-week deltas with no prior week — the most obvious fake-demo tell in the app.
- **How:** Compute real deltas vs the trailing 7-day average from stored CheckIns, or hide the trend pills and show 'Not enough data yet' until >=2 weeks exist. Replace the '+0' domain trends the same way.
### Light up the profile Achievements badges from real thresholds
- **Why:** profile.html:391-396 is six static identical chips (7-Day Streak, 100 Check-Ins, etc.); the .badge-c.earned style exists but is never applied, so hitting a real milestone produces zero unlock dopamine.
- **How:** Bind each tile to values already in context (u.streak, u.total_checkins, stats.domains) and apply .earned when crossed, dim/lock the rest. Makes streaks and check-ins visibly accrue to something.
### Fix account deletion to erase DirectMessages
- **Why:** account_delete (app.py ~2561) deletes 15 models but NOT DirectMessage, so a deleted user's most sensitive 1:1 content persists verbatim in other inboxes, orphaned to a dead user_id. 'Delete my account' currently doesn't delete the private messages.
- **How:** Add DirectMessage (both sender_id and recipient_id) to the delete loop and tombstone the counterpart thread as 'Deleted user'. Straightforward correctness + privacy fix, S effort.
### Add Postgres pool_pre_ping / pool_recycle
- **Why:** app.py:24-25 sets only the URI, no SQLALCHEMY_ENGINE_OPTIONS. Railway reaps idle connections between sparse F&F sessions, surfacing as intermittent 'server closed the connection' 500s. This is a real production bug today, not a future one.
- **How:** Set SQLALCHEMY_ENGINE_OPTIONS = {'pool_pre_ping': True, 'pool_recycle': 280, 'pool_size': 5, 'max_overflow': 5}. One-line reliability fix.
### Defer avatar/banner blob columns off the hot path
- **Why:** current_user() does session.get(User) loading full rows including avatar/banner base64 TEXT on every authenticated page, even though those are served separately via /avatar and /banner. serialize_posts loads full author rows per post just for name + bool(avatar).
- **How:** Mark avatar/banner (and Post.image) as sqlalchemy.orm.deferred and add a cheap has_avatar boolean so bool(a.avatar) doesn't force the blob. Near-free; removes megabytes of base64 from every page and feed render.
### Capture Pro/creator intent instead of toasting it away
- **Why:** twin_pro.html's START PRO button (proSoon) only flashes a toast, and apply_pro.html's 4-step application never POSTs — the /apply-pro route is GET-only with no model. Every high-intent click and every creator application is discarded, so you have no warm-lead list and can't approve a creator.
- **How:** Point START PRO at a POST that sets a pro_waitlist boolean/timestamp on User ('You're on the Founder list'). Add a POST /apply-pro + a ProApplication model (reuse the verification-apply pattern) storing the fields + file, surfaced in admin. S/M effort, real conversion signal during beta.
### Remove the 650ms fake skeleton delay and wire the dead page-transition splash
- **Why:** grow.html (~686) hides fully server-rendered content behind a fixed 650ms skeleton timer, injecting fake latency. Separately base.html's #pt splash overlay + .pt.on CSS exist but nothing ever adds .on, so every nav is a blank reload flash.
- **How:** Drop the setTimeout (show content immediately, keep skeleton markup for the future async fetch). Add a delegated click handler on internal <a> links that adds .pt.on before navigation (skip #, target=_blank, JS buttons). Two small changes that make it feel like an app.

## Pre-beta must-dos
### Streak-at-risk return hero + daily Resend evening email
- **Why:** The bell-dot/golden-pop are driven only by notif_summary total (followers+comments+likes+views, app.py ~1357), all near-zero in F&F, so nothing ever pulls a user back. PWA/push is deferred, but Resend email is already fully wired.
- **How:** On Home, when the user hasn't checked in today but streak is alive, render an urgent hero from the already-computed `shielded` + u.streak. Add a once-daily scheduled job emailing at-risk users at a slightly variable evening time ('Your 12-day streak ends tonight'), reusing send_email. Seeds the future Daily Twin Drop cadence.
### Make the reward loop honest: quests bank XP, spin comes from actions, XP escalates
- **Why:** quests_ctx computes progress but awards nothing, while notifications promise '+X XP' that's never added (app.py:1328). api_spin hands out 3 free spins/day despite UI copy saying spins are earned, and its 'Streak Freeze' prize stores nothing (grace=1 is automatic anyway). Every '+50 XP' check-in is identical on day 1 and day 100. The whole reward economy is decorative or dishonest.
- **How:** Persist quest XP to bonus_xp once per period (track claimed keys on User). Grant spin tokens FROM target actions (check-in, quest complete, first post) instead of a flat daily cap. Add a streak/milestone multiplier to check-in XP using the existing MILESTONES. Either make Streak Freeze a real stored inventory item or remove it.
### Trust & safety bundle for a beta that opens up
- **Why:** Block/mute is shown as 'Soon' but doesn't exist (no BlockedUser model, no /block route); api_dm_send lets anyone message any user_id unlimited with no privacy control, rate limit, or report-from-thread; report calls hardcode 'Reported from feed' with no reason/dedup and land in a flat admin list with no remove/suspend action. One bad friend-of-a-friend has zero recourse.
- **How:** Add BlockedUser(blocker_id, blocked_id) + /api/block toggle, wire the settings row and a Block/Report button on /u/ profiles and DM headers, enforce in api_dm_send/feed/follow. Add a 'who can message me' setting + a simple new-conversation rate limit. Add a report reason sheet + dedup by (user_id, target) + an admin moderation queue filtered to content_reports with remove-post/suspend actions.
### Make the Twin Score move, and let the evening actually change it
- **Why:** twin_score is today-only and ignores ALL checkout data (day_rating, goal_hit change nothing), and domain trends are hardcoded '+0'. A score that never moves reads as a gauge, not feedback — nothing about closing your day matters.
- **How:** Compute a real delta vs the trailing 7-day average and show '▲ +6 vs your week'; fold evening CheckIn.day_rating/goal_hit into the composite so checkout nudges the score; replace the '+0' domain trends with real week-over-week deltas.
### Wire 'HEAR THE TRUTH ABOUT YOU' to real (device) TTS
- **Why:** The loop's emotional peak — the gold button on both success overlays — is vaporware (playMotivation just shows a 'PRO · SOON' card + fake waveform), yet axon_checkin_coach already produces a data-specific message and /api/axon/tts already does device/ElevenLabs TTS.
- **How:** Speak a short 15-30s AXON truth built from today's check-in via the existing checkin-coach text → device TTS (free). Gate ElevenLabs voice and any longer 'speech' to Pro to avoid cost. Also persist the coach reply as an AxonMessage so it shows in /axon history instead of evaporating on page close.
### Cap AXON API spend and add basic error logging before real users hit it
- **Why:** MISSED by the 8 audits: AXON_TIERS_ENABLED=0 means the 15/60-message paywall is OFF — there is no per-user cap on requests.post calls to Anthropic, so a single looping tester (or an abusive link) is an uncapped bill. Combined with swallowed migration exceptions and Railway connection-drop 500s, a solo founder is flying blind on both cost and errors.
- **How:** Turn on a hard daily per-user AXON message cap even in beta (the counts already exist in axon_usage), add a timeout + try/except around the Anthropic requests.post, and add minimal logging (or Sentry free tier) so the intermittent 500s and any failed migration are visible instead of silent.
### Add CSRF protection to state-changing session POSTs
- **Why:** MISSED by all 8 audits including trust: the app is session-cookie auth with POST routes for follow, DM send, block, post create, and account delete. Without CSRF tokens these are forgeable by any page a logged-in user visits — account deletion and DM-send via CSRF are the worst cases. This is a genuine security gap, not a nice-to-have.
- **How:** Add Flask-WTF CSRFProtect (or a manual per-session token checked on POST) and include the token in the existing fetch/form calls. Low effort relative to the risk; do it before the beta widens beyond people you trust.
### Unwall shared /u/ profiles and fix the broken share plumbing
- **Why:** /u/<username> redirects logged-out visitors to /login (app.py:2140), so 100% of shared profile links dead-end at a wall and no share can ever convert. Feed-card share icons (grow.html:283/317) have no onclick, otherShare shares a generic /grow link, and base.html has zero OG tags so every pasted link is a bare grey URL in WhatsApp/iMessage.
- **How:** Render a read-only public profile for logged-out visitors with a sticky 'Join TWIN to follow @user' CTA. Add a /p/<id> per-post route and wire both share affordances to navigator.share with that URL + caption. Add OG/Twitter meta to base.html (one static branded image is enough pre-beta) and a 'Share my TWIN' button that generates the Daily Twin Card from the existing twin_score object.
### Truthful privacy copy + data export before collecting more sensitive data
- **Why:** personal.html says private data (religion, sex, body, quit-habits like 'Porn'/'Alcohol') is 'never shown to anyone / only AXON uses it', but axon_system_prompt injects age/religion into a prompt POSTed to Anthropic — a third party. 'Download my data' is a dead 'Soon' row. This is special-category data with a false promise and no export.
- **How:** Rewrite the microcopy on personal.html/privacy.html: 'AXON is powered by Anthropic; your data is sent to their API to generate coaching and is not used to train models.' Add a /api/export-data route dumping the user's rows (Profile, CheckIns, Habits, Posts, AxonMessages, DirectMessages) as JSON/zip. Cheap now while the schema is small. Also require a real birth_date at signup (reject None) so the 16+ gate can't be skipped by leaving DOB blank.

## Foundation (before trackers)
### Design observations as an append-only event table + daily rollups + a scheduled worker
- **Why:** No observations table exists; CheckIn is the only time-series model and stats_ctx full-scans all of a user's rows. Wearables push thousands of rows/user/day — the current per-page-scan pattern will collapse. This is the actual core product schema and there is no async worker/cron in the app at all today.
- **How:** Single append-only table: id, user_id, source, metric, value_num, value_text, unit, ts (UTC), meta JSONB, with a composite index (user_id, metric, ts DESC). Never read raw per page — add a daily_rollups table (user_id, date, metric, avg/min/max/sum) that twin_score/analytics/feed read from, populated by a Railway cron or RQ worker (also needed for the evening-email job and roll_axon_memory).
### Adopt Flask-Migrate/Alembic before the tracker tables land
- **Why:** Startup runs 24 hand-written 'ALTER TABLE ... ADD COLUMN IF NOT EXISTS' statements every boot (app.py:856-878), each wrapped in try/except that swallows ALL exceptions (883-884) — a genuinely failed migration looks identical to a no-op, so schema drift ships silently. The observations work adds many tables, indexes, and JSONB columns this pattern can't safely manage.
- **How:** Introduce Alembic with versioned migration files and delete the hand-rolled ALTER block. Do this before the data layer so the core product tables have ordering guarantees, down-migrations, and a record of what ran.
### Move base64 images out of DB rows into the Railway Volume / object storage
- **Why:** User.avatar, User.banner, and Post.image are base64 data URLs stored inside primary table rows. This forces Postgres TOAST off-page storage, bloats every backup and any future replication, and makes every join the data layer adds drag megabytes. It gets far worse once observations/rollup tables multiply table sizes.
- **How:** Persist uploads as files on the existing Railway Volume (or S3/R2), store only a path/key in the DB, keep the same /avatar,/banner,/post-image serving routes but stream the file instead of b64decoding a column. Pairs with the deferred-columns quick win.
### Memoize per-request check-in queries and kill the feed N+1
- **Why:** _checkin_dates runs a DISTINCT query called 4-6 times per home render (user_ctx, twin_score→today_metrics, _streaks, streak_shielded, stats_ctx all recompute it), and serialize_posts runs Like.count() + Comment.count() per post despite rank_posts already bulk-computing those counts. Invisible today, but every one of these scans will sit on top of the much larger rollup data, and the feed N+1 scales with post count.
- **How:** Memoize _checkin_dates/_streaks on flask.g so Home computes them once. Pass the bulk like/comment count dicts from rank_posts into serialize_posts instead of re-counting (removes 2N queries per feed render). Cheap now, painful to retrofit.
### Split the 3052-line app.py into models.py + Flask blueprints
- **Why:** One 153KB app.py holds models, ~40 helpers, migrations, and ~90 routes — ~6x over the repo's own 500-line CLAUDE.md rule. Trackers add per-source ingestion parsers, rollup logic, new models, and webhook endpoints (easily 2000+ lines); merging that into the monolith makes every edit a full-file rewrite and makes the observations schema impossible to reason about in isolation.
- **How:** Extract models.py (all db.Model classes + the bootstrap block) and split routes into blueprints (auth, social/feed, axon, checkins, admin, and a new trackers blueprint). Purely structural. Do it after the pre-beta wiring wins but before Phase 3 — and not in the same week as beta launch, since QA just passed on the current layout.

## Later / nice-to-have
- **Weekly TWIN Wrapped recap card** — A repeatable share moment (days checked in, best streak, avg score, top mood, level gained) reusing data already computed in public_profile_ctx and _streaks. Proven recurring virality (Spotify/Duolingo), but lower urgency than fixing the broken per-post/profile share plumbing — schedule it after the core loop and share infra work.

- **Rank 'People you may know' by the follow graph** — people_suggestions just orders by newest signup, ignoring the social graph. Ranking by 2nd-degree overlap ('Followed by @friend') concentrates a small network into visible clusters. Worth doing but the F&F graph is too thin for it to matter until you have more than a couple dozen users.
- **Phase-3 growth mechanics: leaderboards, Daily Twin Drop, FOMO gate, PWA + web push, infinite feed, voiced-AXON free tier** — These are the roadmap's known engagement layer. They depend on the data layer, a real audience, and (for push) PWA infra that's deferred. The evening Resend email and Twin Score movement are the pre-beta stand-ins that seed these; don't pull them forward.
- **Tab-pane fade transitions** — Grow and Profile in-page tabs hard-cut with no animation while adjacent scroll content animates smoothly. Genuine polish inconsistency but the lowest-impact item in the whole audit; batch it with other motion polish long after beta.

## Cut / risky (do NOT do yet)
- **Taking real money (Founder Lifetime / Stripe) during F&F beta** — The money plan wants a $199 lifetime, but you have no entity finalized (QSTP incorporation is still a to-do), no refund/tax/VAT handling, and legality is explicitly deferred until after beta. Charging friends creates chargeback, tax, and trust liability for a handful of sales. Capture Pro-intent (waitlist flag on the START PRO button) now; only turn on Stripe Payment Links once incorporation lands.
- **Building the observations/rollup data layer before the loop retains anyone** — Findings correctly spec an append-only observations table + daily_rollups + a scheduled worker — but that's Phase-3 plumbing. Home is still an empty vessel that discards its own computed Twin Score. Shipping a data warehouse before the existing check-in loop visibly pays off is over-building; you'd have a beautiful pipe feeding a screen no one returns to.
- **Over-investing in the two-sided referral engine at F&F scale** — Referral is marked L-effort. Network effects don't compound at ~a dozen testers. Do the cheap share plumbing (unwall /u/ profiles, per-post URLs, one OG image, the shareable Twin Card) but don't build ref_code attribution + dual-side XP/badge rewards until you have a public launch to compound.
- **Repositioning free-vs-Pro around data-fusion now, then again at Phase 3** — The 'stop giving the moat away free' point is right in spirit, but enforcement of fused-domain gating can't exist until trackers exist. Rewrite the pricing page copy once, lightly (annual as hero, lifetime as anchor). Don't churn the whole free/Pro matrix pre-beta when the features it gates aren't built.
- **Doing the app.py blueprint split + Alembic migration + image-storage move all at once right before beta** — All three are correct foundation work, but QA just passed 28/28 on the current monolith. Bundling three high-surface refactors immediately before F&F risks regressing a stable app. Sequence them: pool config + deferred columns first (near-zero risk), then Alembic, then the split, then image storage — and ideally after beta feedback, not the week before.
- **Wiring 'HEAR THE TRUTH' to multi-minute ElevenLabs voice for free users** — The button should become real via the existing checkin-coach→TTS pipeline, but keep the free tier on device TTS and short (15-30s). AXON tiers are currently disabled, so an uncapped ElevenLabs call on every check-in is a direct cost leak.

## Appendix — all 58 raw findings by dimension

### retention
- **Fire the return hook on streak-at-risk, not just social events** (adjustment, impact=high, effort=M, phase=pre-beta)
  - now: home.html's golden notif-pop and bell-dot are driven only by notif_summary()['total'], which sums followers+comments+likes+views (app.py ~1357). In a friends-and-family beta those are near-zero, so nothing ever pulls a user back. Meanwhile streak_shielded(cu) IS computed and passed to home.html as `shielded`, but the template never references it (dead var), and the .streak-card/.streak-dots CSS is defined but never rendered.
  - fix: Add a self-generated return trigger: when the user has NOT checked in today but their streak is alive, render an urgent hero/pop state ('Your 12-day streak ends at midnight — check in to keep it') using the already-computed `shielded` + `u.streak`. This is the single highest-leverage retention mechanic and the data already exists server-side.
- **Make quest completion actually bank XP (it's currently decorative)** (adjustment, impact=high, effort=M, phase=pre-beta)
  - now: quests_ctx() only computes progress/done flags; nothing awards XP. XP is purely total_checkins*50 + bonus_xp (app.py 1124). Worse, notifications_ctx builds 'Daily quest complete — … (+X XP)' messages (line 1328) promising XP that is never added to bonus_xp. Quests look like a reward loop but pay nothing.
  - fix: On quest completion, persist the quest's XP to bonus_xp once per period (track claimed quest keys per period on the User row). Turn the existing quest UI + '+X XP' notifications into a real, honest reward loop instead of decoration.
- **Wire the spin to an EARNED-action loop and remove the fake Streak Freeze prize** (adjustment, impact=high, effort=M, phase=pre-beta)
  - now: api_spin() (app.py 2947) hands out 3 free spins/day unconditionally, yet the UI copy claims 'a verified action earns a spin' and 'Do 10 push-ups or check in again to earn a spin' (profile.html 285). The action→reward loop that actually forms habits isn't wired. Separately, SPIN_PRIZES includes 'Streak Freeze' ('Your streak's protected') but nothing is stored — grace=1 in _streaks() is automatic regardless, so the prize is a no-op/dishonest.
  - fix: Grant spin tokens FROM target actions (check-in, quest complete, first post) rather than a flat daily cap, so the variable payout reinforces the behavior. Either make Streak Freeze a real stored inventory item that extends _streaks grace, or replace it with a payout that means something.
- **Render the week-streak grid on home for daily loss-aversion** (addition, impact=high, effort=S, phase=now)
  - now: stats_ctx() already produces s['days'] and s['checked'] (a 7-item Sun→Sat boolean of check-ins), and home.html even ships .week-bars/.wb CSS — but the home content block never renders it. The only streak surface is a bare flame+number in the greet-bar with no sense of an unbroken row to protect.
  - fix: Render a 7-dot 'this week' row on home from the existing s['checked'] data, with today's slot empty and pulsing until checked in. Zero new data, pure 'don't break the chain' loss-aversion, reusing CSS that's already in the file.
- **Make the profile Achievements badges reflect real thresholds so milestones visibly unlock** (adjustment, impact=medium, effort=S, phase=now)
  - now: profile.html's STATS 'Achievements' row (lines 391-396: 7-Day Streak, Early Riser, Reader, 30-Day Streak, 100 Check-Ins, All Domains) is static HTML — every tile looks identical whether you're on day 1 or day 100. The PRO 'AWARDS' tab is a permanent empty state. So hitting a real milestone produces no unlock dopamine.
  - fix: Drive these tiles from real values already in context (u.streak, u.total_checkins, stats.domains) with locked vs earned styling (.badge-c.earned already exists), so crossing 7/30/100 lights the badge gold. Cheap, and it's the payoff that makes streaks and check-ins feel like they accrue to something.
- **Ship an interim evening reminder via the existing Resend integration** (addition, impact=high, effort=M, phase=pre-beta)
  - now: There is no channel to pull a user back when the app is closed — PWA/web-push is deferred to Phase 3. But Resend email is already fully wired (send_email/Resend in app.py ~372) and used for signup codes.
  - fix: Add a once-daily scheduled job that emails users who haven't checked in and whose streak is at risk ('Your 12-day streak ends tonight'). It reuses infra you already have and is the only mechanic that makes someone open the app unprompted before push exists. Send at a slightly variable evening time to seed the future Daily Twin Drop cadence.
- **Escalate check-in XP with streak length instead of a flat +50** (adjustment, impact=medium, effort=S, phase=pre-beta)
  - now: checkin_reward() and user_ctx() award exactly +50 XP per check-in day regardless of streak (app.py 1124, 1806). A day-100 check-in feels identical to day 1, so longer streaks give diminishing psychological return and XP has no surprise/variable component.
  - fix: Apply a streak multiplier or milestone bonus to the check-in XP (e.g. +50 base, small bump every N days, big bonus on the 3/7/14/30 milestones already in MILESTONES) so escalating investment makes deep streaks feel increasingly worth protecting.

### coreloop
- **Render the full Twin Score card on home — its verdict + action are computed but invisible** (adjustment, impact=high, effort=S, phase=now)
  - now: twin_score(cu) (app.py:1725) computes score, tier, color, a named verdict, a weakest-lever `action`, and a ring `offset`.
